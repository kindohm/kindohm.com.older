--- layout: post title: "Plight of the Open Source Windows JavaScript Developer" ---

If you are a Windows developer and want to contribute to open source JavaScript projects, prepare for a little bit of pain and suffering.

"Why the pain and suffering?"

Why? Because the Linux and Mac kids are having all the fun right now. They took the initiative and you have to follow in their footsteps. They aren't using Windows tooling (duh!). They are developing their own tooling on node.js or using tools that depend on bash.

Most Windows devs don't have much exposure to node.js, let alone bash.

"What tooling are you talking about?"

The most Windows-unfriendly tooling is the combination of Make and node.js libraries. Separately they are not a problem on Windows, but it isn't so easy (or even possible) to get them to work together on Windows.

The good open source JavaScript projects out there do not use this combination and instead only depend on node.js. However there are still projects out there that use this combination.

In the Linux and Mac world, Make typically runs bash commands to do things like file system manipulation, run tests, and concatenate your JavaScript files. The problem is that Windows can't run bash commands.

One solution is to run Cygwin - which is a Unix emulator for Windows that will run bash. For vanilla Make tasks, this will work just fine. However this becomes a problem when you want to run node.js commands from a Makefile.

"Whoa whoa whoa... can you back up and explain node.js a little?"

In the JavaScript developer world, node.js is exploding. node.js is a JavaScript runtime that abstracts away system details so you can focus on writing JavaScript apps for your system. For the .NET devs out there, think of it this way: node.js is to JavaScript as the .NET Framework is to C#.

The best JavaScript tooling is being built on top of node.js. Take for example two popular node.js libraries: UglifyJS and grunt. UglifyJS is arguably the fastest and most valid JavaScript minifier in existence today. Many JavaScript projects are using it. Grunt is "a task-based command line build tool for JavaScript projects". Use Grunt to structure your JavaScript projects and carry out Make-like build tasks. These two libraries are run in the node.js runtime.

These two libraries are just a taste. There are hundreds of JavaScript libraries built on node.js that make your own JavaScript development more productive.

I should mention that node.js installs and runs on Windows just fine. You can even alias it in PowerShell.

"Ok, so what's the problem with Cygwin and node.js?"

Some open source JavaScript projects use a combination of both Make and node.js. For example, they use Make to fire off UglifyJS minification. On Windows, this would mean you'd need to get node.js running on Cygwin. However, if you want to try and get node.js running on Cygwin, you are in for a world of hurt: github.com/joyent/node/wiki/Building-node.js-on-Cygwin-(Windows).

Simply put, node.js isn't going to be supported by Joyent to run on Cygwin anytime soon.

"So I should get to know node.js on Windows, and for a new JavaScript project I should use only node.js libraries for build and dev tasks. Easy enough."

Yep.

"But since I develop on Windows I can't contribute to projects that depend on both Make and node.js?"

Because we're talking about open source, the answer is no!!!

The real task at hand is to eliminate the dependency on Make and to roll with 100% node.js tooling for JavaScript projects. Grunt does what Make was already doing in JavaScript builds, so eliminate the dependency on Make.

Leverage the fact that this is open source. Contribute to projects to help eliminate the bash and Make dependencies. The owners of the projects will most likely be delighted that someone is willing to contribute and improve the code base; that has at least been my experience with open source projects.

"Wow, that doesn't sound too painful."

It isn't, but it's a different world for the average Windows .NET developer who spends most of their time in Visual Studio. If you don't have much experience working from a command line then prepare for a different learning curve. Unless you want to dig in deep and spend a lot of extra time developing some plugins, you are going to get your hands dirty in the command line with node.js. In my opinion, this is a good thing.

For straight-up JavaScript development, big IDE's aren't going to give you much. JavaScript doesn't use a compiler. There is no need for Solution, Project, Resource, or UI meta-files. Keeping it real with the command line and a good text editor are what it's about with JavaScript. However, arguing about a programmer's preference in editors is like religious warfare. The bottom line is to use what makes you most productive, whether that's Notepad, Notepad++, Notepad2, Vim on Windows, or an IDE like Visual Studio [1].

If you are a Windows JavaScript developer, or a Windows developer at all for that matter, get to know node.js on Windows. Install it and start using it. Develop for it. It is how JavaScript development on Windows should be done, especially if it is for an open source project. It levels the playing field for JavaScript development across all of the main operating systems.

[1] Personally I have fun using tmux to host a Vim editor and additional console prompts to run builds and tests. You never leave the keyboard and it's faster than Alt+Tab'ing since it's only one Window. However, tmux is not supported in Cygwin, so you need to dev on a Linux or Mac machine to go this route. Yet another pain for the Windows developer...if you want to go the tmux route.
